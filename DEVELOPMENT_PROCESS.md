# 🚀 프로젝트 개발 및 배포 표준 프로세스 (v1.0)

본 문서는 프로젝트의 안정성과 품질 관리를 위해 모든 개발 작업에 강제되는 표준 워크플로우를 정의합니다. 모든 팀원 및 AI 어시스턴트는 본 프로세스를 엄격히 준수해야 합니다.

## 📋 표준 워크플로우 (6단계)

1.  **[Step 1] 작업 오더 (Job Order)**
    *   사용자(CTO)로부터 작업 목표 및 요구사항을 명확히 전달받습니다.
    *   요구사항이 모호할 경우 질문을 통해 명확히 정의합니다.

2.  **[Step 2] 역할별 작업 수행 (Work)**
    *   정의된 역할에 맞춰 코드를 수정하거나 기능을 구현합니다.
    *   이 단계에서는 로컬 환경에서의 파일 수정만 이루어집니다.

3.  **[Step 3] 코드 검토 (Self Code Review)**
    *   수정된 코드를 스스로 검토합니다.
    *   `npm run build` 등을 통해 구문 오류, 빌드 오류, 린트 에러를 체크합니다.
    *   로컬 개발 서버에서 기능 동작 및 UI/UX를 자체 검증합니다.

4.  **[Step 4] CTO 최종 검토 (CTO Final Review)**
    *   작업 결과를 사용자(CTO)에게 상세히 보고합니다.
    *   수정된 파일 목록, 변경 내용의 핵심, 로컬 테스트 결과 등을 공유합니다.
    *   **※ 주의**: 이 단계에서 오류가 발견되거나 요구사항이 미달성되었을 경우 [Step 2]로 돌아가 수정 작업을 진행합니다.

5.  **[Step 5] 최종 확인 (Final Confirmation)**
    *   사용자(CTO)가 결과물을 확인하고 최종 승인(Approval)을 내립니다.

6.  **[Step 6] 서버 반영 (Server Deployment)**
    *   사용자의 최종 승인이 있는 경우에만 `git push`를 실행하여 서버에 코드를 반영합니다.
    *   반영 후 정상 배포 여부를 최종 확인합니다.

---

## ⚠️ 강제 사항 (Compliance)
*   사용자의 **명시적 승인** 없이 `git push`를 실행하는 행위는 금지됩니다.
*   모든 배포는 로컬 빌드 성공(Exit Code 0)이 확인된 후에만 가능합니다.
*   작업의 각 단계가 완료될 때마다 현재 진행 상황을 명확히 보고합니다.
